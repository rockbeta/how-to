LLM agent mechanics

How LLM-Based Agents Work

A coding agent is a control loop around a language model. The model reads structured messages, chooses an action, calls a tool, observes the result, then generates or revises code until verification passes.

Follow a sample run See a run

The LLM predicts the next useful message from the context it sees: system rules, developer instructions, user prompt, file excerpts, tool schemas, previous observations, and any compact state the agent runtime keeps.

The coding agent runtime turns those messages into action. It selects tools, enforces permissions, applies patches, runs checks, feeds results back to the model, and decides whether to continue, ask, or deliver.

A real agent does not simply emit final code. It exchanges typed packets with the runtime: prompt messages, compact reasoning state, tool calls, observations, edits, and verification results.

Runtime loop

The model and runtime take turns.

Each turn adds evidence to the transcript. The runtime handles effects; the model handles interpretation, selection, and synthesis.

Data surfaces

A coding agent is a protocol around an LLM.

01

Prompt Assembly

Packs system rules, developer guidance, user intent, retrieved snippets, and recent observations into the model context.

02

Context Budget

Decides which files, logs, summaries, and prior decisions fit in the context window for the next model call.

03

Tool Schema

Describes callable operations and their argument shapes, such as reading files, applying patches, running tests, or opening a page.

04

Sandbox

Separates harmless inspection from writes, network access, and commands that need explicit permission.

05

Patch Layer

Converts model-proposed edits into deterministic file changes, usually as structured updates or unified diffs.

06

Verifier

Runs tests, linters, screenshots, type checks, or review passes and returns compact evidence for the next loop.

Coding-agent trace

One request becomes a sequence of typed turns.

The visible answer is only the final turn. The useful work is the repeated exchange of actions and evidence.

  1. Goal

    User asks: add a technical explainer for the agent loop.

  2. Context

    Runtime sends current files, repo state, and tool schemas to the model.

  3. Decide

    Model selects a small edit plan and asks to inspect relevant files.

  4. Patch

    Runtime applies HTML, CSS, and JavaScript edits proposed by the model.

  5. Verify

    Browser and diff checks return evidence: layout, interactions, errors.

  6. Deliver

    Agent summarizes changed files, checks run, and remaining caveats.
