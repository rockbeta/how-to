How KV Cache Works in LLMs

KV cache is the memory structure that lets a transformer reuse the keys and values computed for previous tokens. It makes generation much faster, but turns context length into a serious memory cost.

In self-attention, every token creates query, key, and value vectors. The query asks what to look for; keys say what each token offers; values carry the information that gets mixed into the current hidden state.

During generation, old keys and values do not change. The runtime stores them once, then each new token only computes its own Q, K, V and attends over the cached K and V from the whole context.

Phase 1

Prompt processing is cache construction.

When LM Studio says it is processing the prompt, the model is doing a full forward pass over the input tokens. This is also called prefill.

What happens in prefill

  1. The chat template formats system, user, and assistant messages.
  2. The tokenizer converts text into token IDs.
  3. Embeddings turn token IDs into vectors.
  4. Each transformer layer computes Q, K, V for every prompt token.
  5. The model uses causal attention so token n can only see tokens before it.
  6. K and V tensors are saved into the KV cache for later decoding.
  7. The final prompt position produces the distribution for the first generated token.

Attention per head

Q = X Wq
K = X Wk
V = X Wv

Attention(Q, K, V) =
  softmax(Q K^T / sqrt(d_head)) V

In prefill, Q, K, and V are built for many prompt tokens at once. The expensive part is comparing prompt tokens with previous prompt tokens across every layer.

Phase 2

Generation reuses old K and V.

Once the prompt cache exists, the model decodes one token at a time. The new token gets a fresh query, key, and value; the old tokens contribute through cached K and V.

Step 1

Prefill prompt tokens

The model processes prompt tokens in parallel and writes K/V vectors for each layer into cache.

Without KV cache

for each generated token:
  run transformer on all tokens so far
  recompute old K and V again
  sample next token

Simple, but wasteful. Old tokens are recomputed every step.

With KV cache

prefill(prompt):
  cache K,V for prompt tokens

decode(new_token):
  compute Q,K,V for new token
  append new K,V to cache
  attend Q over cached K,V
  sample next token

Fast per token, but cache memory grows with sequence length.

Memory model

The cache is large because it stores every layer.

KV cache size scales linearly with batch size, context length, layers, KV heads, head dimension, and bytes per cache element.

Shape

K_cache[layer][batch][kv_head][token][head_dim]
V_cache[layer][batch][kv_head][token][head_dim]

Approximate bytes

2 * layers * batch * kv_heads
  * seq_len * head_dim * bytes_per_value

The leading 2 is for K and V. Some runtimes store the dimensions in a different order for speed, but the count is the same.

Inference systems

KV cache explains many runtime behaviors.

01

Long prompts feel slow

Prefill must process all prompt tokens and create cache entries for every layer. A longer chat history means more prompt work.

02

First token latency

The first output token waits for prompt tokenization, prefill, memory allocation, and sometimes GPU warmup.

03

GQA and MQA

Grouped-query and multi-query attention use fewer KV heads than query heads, reducing cache memory and bandwidth.

04

Prompt caching

If a prefix is reused exactly, some systems can reuse its KV cache and avoid reprocessing that prefix.

05

Paged attention

Serving engines often store cache in blocks, like virtual memory, so many requests can share GPU memory without large contiguous allocations.

06

Sliding windows

Some models attend only to a recent window, allowing the runtime to evict older cache entries and cap memory growth.

Mental model

Why the cache matters.

The prompt is read once during prefill. Its keys and values become reusable memory. Each generated token adds one more slice to that memory, then uses a fresh query to look back over it.

That is why local inference has two different speeds: prompt processing throughput for prefill, and tokens-per-second for decode. KV cache is the bridge between them.
